# v0.4.1 Ergonomics Plan

> Sits ON TOP OF v0.4.0 (T4's work: machine-readable errors + PcsError). Must
> be executed AFTER T4 lands and the `PcsError` taxonomy is available. Items
> use the error codes T4 defines (e.g. throw `PcsError("PCS_STALE_SESSION", …)`).

**Goal:** close the four ergonomics gaps surfaced during field-testing by
Marty (independent Claude Code session, 2026-04-18), on top of v0.4.0.

## The four items

### A. Session staleness warning on use

**Problem:** `list` only probes when asked. A saved session can go DEAD
(cookie expired, server-side invalidation) between `list` runs. A Claude
session reaching for `--session=<name>` tomorrow may silently hit a 302
and produce a meaningless screenshot or an auth wall.

**Design:**
- When any browser command resolves `--session=<name>`, check the probe
  cache (`~/.playwright-sessions/.probe-cache.json`).
- If last probe for that session is **>6h old** OR missing: run a single
  synchronous fast HEAD probe (<1s budget) against the session's primary
  service URL before launching the browser.
- If probe says DEAD: throw `PcsError("PCS_STALE_SESSION", …)` with
  `suggest="playwright-cli-sessions refresh <name>"`. Exit 77 (same class
  as PCS_AUTH_EXPIRED).
- If probe says LIVE: update cache, proceed.
- Opt-out: `--no-probe` flag OR `PLAYWRIGHT_CLI_NO_STALE_CHECK=1`.

**Files:**
- `src/store.ts` — new `checkSessionFreshness(name)` function
- `src/commands/*.ts` — call freshness check in the hydration path
  (shared path — probably belongs in `browser-launch.ts` or a new
  `src/session-use.ts`)
- `tests/scenarios/session-staleness.sh` — new scenario, fixture-based

### B. HTTP-error detection on navigation

**Problem:** `screenshot https://vercel.com/some-404-url` returns exit 0
with a screenshot of the 404 page. No way to tell at the shell level
that the page is an error page vs. the real content.

**Design:**
- When `navigate`/`screenshot`/`snapshot`/`expect` navigate, capture the
  initial response status via `response.status()` on `page.goto()` return.
- If status is 4xx or 5xx: throw `PcsError("PCS_HTTP_ERROR", …)` with
  `{ status, url, finalUrl }`. Exit code 11 (same as PCS_NAV_FAILED).
- Opt-out: `--allow-http-error` flag (e.g. if the user is intentionally
  screenshotting an error page) OR `PLAYWRIGHT_CLI_ALLOW_HTTP_ERROR=1`.
- **Exception:** auth-wall detection takes priority — if status is 401 or
  302-to-login, it's `PCS_AUTH_WALL`, not `PCS_HTTP_ERROR`.

**Files:**
- `src/browser-launch.ts` or new `src/http-guard.ts`
- Each navigation command — integrate the guard
- `tests/scenarios/http-errors.sh` — new

### C. Better wait primitives

**Problem:** `--wait-for=<selector>` is blunt. For SPAs, waiting for a
selector that's always present (e.g. `body`) doesn't prove the data has
loaded. Users end up writing `exec` scripts just to chain waits.

**Design — three new flags, each thin aliases:**
- `--wait-for-text="<substring>"` — wait until page text contains the
  substring. Implemented as `page.waitForFunction(() => document.body.innerText.includes(…))`.
- `--wait-for-count=<selector>:<N>` — wait until `document.querySelectorAll(selector).length >= N`.
  Useful for waiting on lists/tables to render.
- `--wait-for-network=idle` — alias for `--wait-until=networkidle`. More
  discoverable name.

All three compose with `--wait-for`/`--wait-until` (AND semantics — all
specified waits must pass).

**Files:**
- `src/browser-launch.ts` — wait-orchestrator helper
- Each navigation command — plumb flags through
- `src/cli.ts` — add flags to whitelists (T4 introduces these)
- `tests/scenarios/waits.sh` — new

### D. `exec --eval='<js>'` and stdin support

**Problem:** Every exploratory `exec` requires writing a `.mjs` file to
disk. Kills iteration speed when the AI is trying to figure out what a
page's DOM looks like.

**Design:**
- `playwright-cli-sessions exec --eval='<js>' [url]` — treat `<js>` as
  the BODY of an async function that receives `{ page, context, browser }`.
  Equivalent to:
  ```
  export async function run(ctx) { <js> }
  ```
- `playwright-cli-sessions exec - [url]` (script path = `-`) — read the
  script from stdin.
- Both paths support the same flags as file-based `exec`.

**Files:**
- `src/commands/exec.ts` — add `--eval` and stdin paths
- `src/cli.ts` — add to flag whitelist
- `tests/scenarios/exec.sh` — extend with 3 new cases
